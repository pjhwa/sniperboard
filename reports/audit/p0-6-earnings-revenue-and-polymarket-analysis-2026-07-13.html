7 Polymarket vs Kalshi — P0-4 대안 분석

한국에서 Kalshi API 키를 쓸 수 없을 때, Polymarket을 FOMC/매크로 확률 소스로 쓸 수 있는지 평가한다.

7.1 요구 기능 (현재 prediction 파이프라인)

  • 다음 FOMC 결정 확률 분포 (동결 / 25bp 인하 / 인상 등)
  • 주기적 수집 → JSON → SniperBoard 표시
  • 가능하면 API 키 없이 또는 한국에서 합법·기술적으로 접근 가능

7.2 비교표

항목Kalshi (현 설계)Polymarket (후보)
규제 미국 CFTC 규제 예측거래소 블록체인 기반, 관할·접근성이 다름 (지역 제한 변동 가능)
한국에서 API 키 사용 불가/어려움 (사용자 제약) 시장 읽기용 공개 HTTP API 존재 (이벤트/마켓 조회에 키 불필요 케이스 다수)
FOMC 유동성 설계상 적합 (이벤트 계약) 실거래 규모 큼 — 예: Fed decision 이벤트 수천만~억 달러 누적 볼륨 보도·집계 존재
참여자 구성 규제 시장 참여자 글로벌·크립토 자본 비중 큼 → 편향 가능
가격 의미 계약 가격 ≈ 확률 동일하게 mid/last ≈ 함의 확률 (단, 스프레드·유동성 구간 주의)
스키마 적합 기존 prediction/latest.json next_fomc 동일 스키마로 매핑 가능 (outcomes + probabilities)
운영 복잡도 API 키·시리즈 티커 유지 이벤트 슬러그 탐색·만료 마켓 교체 자동화 필요
SniperBoard 적합성 본래 의도 참고용 매크로 카드로는 적합, “공식 확률” 표현은 부적합

7.3 Polymarket 데이터 “맞는가?” — 분석 기준

기준평가설명
정보 가치 높음 실시간 머니 배팅 확률은 서베이보다 빠른 경우가 많음. FOMC 전후 반응 관찰에 유용.
통계적 공정성 중간 참여자 ≠ 미국 채권/주식 기관 전체. 크립토 레짐 편향·헤지 수요 가능.
재현성·자동화 가능 공개 REST(이벤트/마켓/가격)로 cron 수집 가능. 키 의존도 낮음.
법·접근 리스크 중간 지역별 접근 정책 변경 가능. 읽기 전용 시세와 거래 참여를 분리해 설계할 것.
제품 메시지 중요 “시장 함의 확률(Polymarket)” vs “연준 공식 전망”을 UI에서 명확히 구분해야 함.
사실 (공개 정보 기준) Polymarket에는 Fed/FOMC 관련 고유동성 마켓이 반복적으로 형성되며, 공개 시세 집계·볼륨 보도가 존재한다. 이는 “데이터가 없다”가 아니라 “성격이 Kalshi와 다른 예측시장 데이터”임을 의미한다.

8 권고: Polymarket을 사용할 것인가?

권고: 조건부 Yes (참고용)

Kalshi가 막힌 환경에서 FOMC 확률 공백(P0-4)을 메우는 1차 대안으로 Polymarket 공개 시세를 쓰는 것은 타당하다. SniperBoard의 매크로/모닝 브리핑에 “시장 함의” 레이어로 붙이면 제품 가치가 있다.

단, 이렇게는 쓰지 말 것
  • Kalshi와 동일 신뢰도로 병합
  • “연준이 X% 확률”처럼 공식 톤 표기
  • Conviction/Stage2 가중치에 직접 편입 (현재 설계 범위 밖)
  • 단일 thin 마켓을 검증 없이 채택

최종 판단 요약

질문답
이 데이터를 사용하는 게 맞나? 맞다 — 단 ‘참고용 예측시장 확률’로 한정할 때. 거시 이벤트 전 포지션 경계·리스크 온/오프 해석에 유용하다.
Kalshi를 완전히 대체하나? 기능 대체(확률 숫자 공급)는 가능, 의미 대체는 불완전. source 필드에 polymarket을 명시하고 UI에 출처 배지 필수.
지금 당장 구현해야 하나? P0-6 완료 후 P0-4 구현 시, Kalshi 복구를 기다리지 말고 Polymarket 어댑터를 우선 구현하는 것을 권장.

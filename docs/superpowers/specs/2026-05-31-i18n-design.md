# i18n Design: English & Korean Multilingual Support

**Date:** 2026-05-31  
**Scope:** `~/dev/market-sentiment-data` + `~/dev/sniperboard`  
**Languages:** English (en) · Korean (ko)

---

## 1. Overview

Three workstreams in parallel:

1. **market-sentiment-data** — AI pipeline generates bilingual JSON fields (`_en` / `_ko`)
2. **sniperboard UI** — Topbar EN/KO toggle, Zustand locale state, inline bilingual objects
3. **Documentation** — Language-per-file strategy: `README.md` (EN), `README.ko.md` (KO)

---

## 2. market-sentiment-data: Schema v2.0

### 2.1 Schema Version Bump

`schema_version`: `"1.4"` → `"2.0"`

### 2.2 Field-Level Changes

All AI-generated human-readable text fields get `_en` / `_ko` suffix variants. The old unsuffixed field is **removed** (SniperBoard is the only consumer and will be updated simultaneously).

**`latest.json` / `history/*.json` — market block:**

| Removed | Added |
|---------|-------|
| `market.key_reason` | `market.key_reason_en`, `market.key_reason_ko` |
| `market.top_news.headline` | `market.top_news.headline_en`, `market.top_news.headline_ko` |
| `market.top_news.summary` | `market.top_news.summary_en`, `market.top_news.summary_ko` |

**`latest.json` / `history/*.json` — symbols[] block:**

| Removed | Added |
|---------|-------|
| `symbols[].key_reason` | `symbols[].key_reason_en`, `symbols[].key_reason_ko` |
| `symbols[].top_news.headline` | `symbols[].top_news.headline_en`, `symbols[].top_news.headline_ko` |
| `symbols[].top_news.summary` | `symbols[].top_news.summary_en`, `symbols[].top_news.summary_ko` |

**`brief/latest.json` / `brief/history/*.json`:**

| Removed | Added |
|---------|-------|
| `market_brief.summary` | `market_brief.summary_en`, `market_brief.summary_ko` |
| `market_brief.watch_points` | `market_brief.watch_points_en`, `market_brief.watch_points_ko` |
| `market_brief.key_themes[]` | `market_brief.key_themes_en[]`, `market_brief.key_themes_ko[]` |
| `symbol_briefs[].brief` | `symbol_briefs[].brief_en`, `symbol_briefs[].brief_ko` |
| `symbol_briefs[].key_risk` | `symbol_briefs[].key_risk_en`, `symbol_briefs[].key_risk_ko` |
| `symbol_briefs[].key_opportunity` | `symbol_briefs[].key_opportunity_en`, `symbol_briefs[].key_opportunity_ko` |

**Non-human-readable fields stay unchanged:** `sentiment`, `sentiment_score`, `slot`, `generated_at`, `schema_version`, `trend_vs_yesterday`, `confidence`, `source`, `divergence`, `intraday_shift`, `composite_score`, `price_context.*`, `setup_quality`, `action_bias`, `tone`.

### 2.3 Example Output (v2.0)

```json
{
  "schema_version": "2.0",
  "market": {
    "key_reason_en": "AI and tech stocks drive S&P 500 to record highs for 9 consecutive weeks.",
    "key_reason_ko": "AI·기술주 집중으로 S&P 500가 9주 연속 사상 최고 기록 중.",
    "top_news": {
      "headline_en": "IT sector now 37% of S&P 500 market cap — all-time high",
      "headline_ko": "IT 섹터 비중 37%로 사상 최고 기록",
      "summary_en": "Technology concentration surpasses dot-com era as AI stocks lead gains.",
      "summary_ko": "기술주 집중이 도트컴 시대를 넘어서며 AI 관련주가 상승 주도."
    }
  }
}
```

### 2.4 Pipeline Change: collect_sentiment.py

Modify the AI prompt (Grok/Hermes call) to instruct the model to return bilingual fields directly in the JSON response:

```
IMPORTANT: For all human-readable text fields (key_reason, headline, summary, brief, key_risk, key_opportunity, watch_points, key_themes), 
provide BOTH an English version (suffix _en) and a Korean version (suffix _ko).
Do NOT include unsuffixed versions of these fields.
```

No separate translation step is needed — the AI generates both in one call.

### 2.5 schema.json Update

- Bump `$schema` version field
- Add `_en` / `_ko` property definitions for all affected fields
- Remove unsuffixed definitions

## 3. SniperBoard UI: i18n Architecture

### 3.1 No External Library

Custom solution using Zustand + inline bilingual objects. No new npm dependency.

### 3.2 New File: `frontend/app/i18n.ts`

```ts
export type Locale = 'en' | 'ko'

export interface BiLang {
  en: string
  ko: string
}

export const t = (obj: BiLang, locale: Locale): string => obj[locale]
```

### 3.3 Zustand Store Change (`frontend/hooks/useStore.ts`)

Add to store state:
```ts
locale: Locale  // default: 'ko' (existing users see no change)
setLocale: (l: Locale) => void
```

Initialization: read from `localStorage.getItem('locale')` on store creation. `setLocale` persists to localStorage.

### 3.4 File Changes

**`frontend/app/types.ts`**

Convert all human-readable label/desc fields from `string` to `BiLang`:

- `REGIME_META[*].label` → `{ en: 'Risk-On', ko: '강세' }`  
  (already has `labelKo` — merge into BiLang and drop `labelKo`)
- `REGIME_META[*].desc` → `{ en: '...', ko: '...' }`
- `DD_META[*].label` → `{ en: 'Normal (0-3d)', ko: '정상 (0~3일)' }`
- `DD_META[*].desc` → `{ en: '...', ko: '...' }`
- `SIGNAL_META[*].label` → stays English (signal names are universal)
- `SIGNAL_META[*].action` → `{ en: 'Entry', ko: '진입' }`
- `SIGNAL_META[*].desc` → `{ en: '...', ko: '...' }`

**`frontend/app/glossary.ts`**

```ts
export interface GlossaryEntry {
  key: string
  term: { en: string; ko: string }
  body: { en: string; ko: string }
}
```

All 28 entries get `term.en`, `term.ko`, `body.en`, `body.ko`.

**Components** — replace hardcoded Korean strings:

| Component | Strings to translate |
|-----------|---------------------|
| `Topbar.tsx` | Board title display, search placeholder |
| `BottomTabs.tsx` | Tab labels (시장/종합분석/매크로/심리) |
| `MarketStrip.tsx` | "? 가이드" button label |
| `CommandPalette.tsx` | Placeholder, "용어 검색 모드" banner, result count text |
| `OverviewBoard.tsx` | Card titles, section headers, freshness text |
| `DeepDiveBoard.tsx` | Row labels, section headers, KPI labels |
| `IntradayBoard.tsx` | Signal section header, RSI label, action bar text |
| `DailyBoard.tsx` | Section headers |
| `MacroBoard.tsx` | Group labels, signal labels |
| `SentimentBoard.tsx` | Section headers |
| `BoardGuidePanel.tsx` | Each board's guide section titles and bodies |

### 3.5 Topbar Language Toggle

Add `EN | KO` toggle to Topbar right side (next to theme toggle):

```tsx
<button onClick={() => setLocale('en')} className={locale === 'en' ? 'active' : ''}>EN</button>
<button onClick={() => setLocale('ko')} className={locale === 'ko' ? 'active' : ''}>KO</button>
```

Styled with existing Plaid DS tokens. Mobile: included in topbar (small text, no overflow).

### 3.6 Data Layer (API Responses)

When the backend serves bilingual brief/sentiment data, the frontend reads the `_en` or `_ko` field based on current locale:

```ts
// Example in DeepDiveBoard
const reason = locale === 'en' ? sentiment.key_reason_en : sentiment.key_reason_ko
```

Backend schemas (`backend/api/schemas.py`) updated to include `_en`/`_ko` fields as `Optional[str]` with graceful fallback.

## 4. Documentation Strategy

### 4.1 File Convention

| Current file | New English file | New Korean file |
|---|---|---|
| `README.md` (KO) | `README.md` (EN, rewritten) | `README.ko.md` (current content) |
| `PROJECT_CONTEXT.md` (KO) | `PROJECT_CONTEXT.md` (EN) | `PROJECT_CONTEXT.ko.md` (current content) |
| `CLAUDE.md` (KO) | `CLAUDE.md` (EN) | `CLAUDE.ko.md` (current content) |
| `schema.json` descriptions | English | — |

### 4.2 README Convention

Each `README.md` (English, primary) links to its Korean counterpart at the top:

```md
> 한국어 문서: [README.ko.md](./README.ko.md)
```

Each `README.ko.md` links back:

```md
> English docs: [README.md](./README.md)
```

### 4.3 Repos Covered

- `~/dev/market-sentiment-data`: README.md, CLAUDE_CODE_INSTRUCTIONS files
- `~/dev/sniperboard`: README.md, PROJECT_CONTEXT.md, CLAUDE.md

  2a. Update backend/api/schemas.py (add _en/_ko Optional fields)

  3a. Create frontend/app/i18n.ts
  3b. Add locale to useStore.ts

  4c. Convert all 7 board components
  4d. Convert BoardGuidePanel sections

Phase 5: Documentation
  5a. sniperboard: README, PROJECT_CONTEXT, CLAUDE bilingual split

---

## 6. Out of Scope

- Chart axis labels, candlestick data — numeric, language-agnostic
- `macro/latest.json` — structure is already key-value with numeric signals; human-readable text handled via `macro_rules.py` labels in types.ts (covered in Phase 3c)
- Translation memory / TMX files — not needed for this scale
- RTL support — Korean/English are both LTR
- Automated translation quality checks

---

## 7. Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| AI generates inconsistent bilingual JSON (missing `_en` or `_ko`) | Add validation in collect script; fall back to other language if one is null |
| Large volume of string changes causes UI regressions | Phase 4 done board-by-board with visual check after each |
| `localStorage` locale not set for new users | Default to `'ko'` to preserve existing UX |
| market-sentiment-data history files still use v1.4 schema | Consumers check `schema_version`; frontend falls back gracefully for old history files |
